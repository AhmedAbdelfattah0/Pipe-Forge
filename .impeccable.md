# PipeForge — Impeccable Design Guide

## Design Context

### Users
DevOps engineers, frontend engineers, and development teams who build Azure DevOps pipelines. They work in fast-paced environments where CI/CD reliability is critical. They come to PipeForge to generate correct, production-ready pipeline configurations without wrestling with Azure Portal's complexity. Their context is task-focused: get in, configure, generate, ship.

### Brand Personality
**Precise. Fast. Trustworthy.**

- **Voice**: Direct and confident. No fluff, no marketing speak. Speak like a senior engineer explaining something clearly.
- **Tone**: Calm authority. The UI should feel like it already knows what you need.
- **Emotional goal**: Users should feel **confident the output is correct** and **relieved it was effortless**. Every interaction should reinforce that PipeForge knows what it's doing.

### Aesthetic Direction
**Visual tone**: Clean, modern developer tool. Minimal surface area, maximum clarity. The interface should feel fast even before it responds.

**References**:
- **Linear** — Clean, fast, precise. The gold standard for developer tool UI.
- **Vercel** — Developer trust through minimalism. Confident use of whitespace.
- **Railway** — Modern devtool aesthetic. Approachable without being playful.

**Anti-references**:
- **Jira** — Cluttered, heavy, visually overwhelming. The opposite of what we want.
- **AWS Console** — Outdated, dense, intimidating. Too many options competing for attention.
- **Azure Portal** — The very thing PipeForge replaces. If it feels like Azure Portal, we've failed.

**Theme**: Light mode only (V1). Dark mode planned for V3 Enterprise tier.

**RTL**: Not required. UI is English-only. Arabic language codes (EN/AR) appear only in generated pipeline output, not in UI direction.

### Design Principles

1. **Clarity over cleverness** — Every element earns its place. No decorative complexity, no ambiguous icons, no jargon without context. If a user has to pause and think about what something means, simplify it.

2. **Speed is a feature** — The interface should feel instant. Minimal visual weight, fast transitions, no unnecessary steps. Reduce clicks, reduce fields, reduce cognitive load. Get users to their pipeline faster.

3. **Confidence through feedback** — Every action should produce clear, immediate feedback. Validation states, progress indicators, and output previews all serve one goal: the user never wonders "did that work?"

4. **Quiet professionalism** — No flashy gradients, no bouncing animations, no emoji in UI copy. The aesthetic is a well-tailored suit, not a billboard. Let the work speak.

5. **Accessible by default** — WCAG 2.1 AA minimum. Focus rings, semantic HTML, proper contrast ratios, and `prefers-reduced-motion` support are not afterthoughts — they're baseline requirements.

### Color System
```
Primary:    #2E86C1  — Actions, active states, focus rings
Navy:       #1E3A5F  — Sidebar, dark panels, headings
Surface:    #F8F9FA  — Light backgrounds
Card:       #FFFFFF  — Cards, form fields
Border:     #E2E8F0  — Dividers, input borders
Text Dark:  #1A1A2E  — Primary body text
Success:    #1E8449  — Positive states (QA, production, confirmations)
Warning:    #D35400  — Caution states (secrets, sensitive fields)
```

### Typography
- **Family**: Inter (Google Fonts), weights 400–700
- **Fallback**: -apple-system, BlinkMacSystemFont, 'Segoe UI', Roboto, sans-serif
- **Smoothing**: Antialiased on all platforms

### Component Prefix
All shared components use the `pf-` prefix (e.g., `pf-button`, `pf-input`, `pf-card`).

### Tech Constraints
- Angular 20, standalone components, signals
- Tailwind CSS v4 with `@theme` design tokens
- Lucide Angular for iconography
- OnPush change detection throughout
- Mobile-first responsive (Tailwind breakpoints: sm/md/lg/xl)
